How to Read This Document
This document is the authoritative API contract between the NESAm Spring Boot backend and the React frontend. It is organised by frontend page rather than by technical layer, because every API exists to serve a specific screen. If a screen is blocked, you look it up here and know exactly what to build.

Document Structure
• One section per frontend page or domain, in the order a user encounters them: Home ? Login ? Register ? Member Dashboard ? Active DFC ? Impact ? Admin Dashboard ? Profile Management ? Admin Configuration.
• Each endpoint card shows: the HTTP method and path, authentication requirement, the SRS requirement it satisfies, the invariants it must enforce, why this API exists in plain English, which frontend ViewModel or hook calls it, the exact request and response field contracts, and any critical notes.
• An error contract section defines the standard error response shape that the frontend Axios interceptor and React Hook Form validation rely on.
• A Master API table at the end gives a one-page summary of all endpoints for quick reference.

Base URL
All paths in this document are relative to the base URL:http://localhost:9090/api/v1Example: POST /public/registration/verify-emailFull URL: http://localhost:9090/api/v1/public/registration/verify-email
Mock Flag
The frontend has a USE_MOCK flag in src/shared/constants/config.ts. While this is true, all API calls return fake data from the mocks/ folder of each domain. Flip it to false one domain at a time as backend endpoints become ready. The component code never changes — only the data source switches.


Error Response Contract
Before building any endpoint, the backend and frontend must agree on the error response shape. The frontend Axios interceptor (shared/api/axiosInstance.ts) and React Hook Form field-level validation both depend on this contract. Implement a global @ControllerAdvice in Spring Boot that produces this format for every error.

Standard Error Payload (RFC 7807)
FieldTypeDescriptiontypestringMachine-readable error category. Use invariant codes where applicable: INV-EL-02, INV-NM-01, etc. Otherwise use descriptive strings like VALIDATION_ERROR.titlestringShort human-readable summary. Example: "Age validation failed".statusintegerHTTP status code repeated in the body. 400, 422, 401, 403, 404, 409, 500.detailstringFull explanation suitable for the user or developer. Example: "Applicant age must be less than 60 years on the date of payment."violationsarrayPresent only on validation errors (422). Each item is { field: string, message: string } where field is the exact DTO field name React Hook Form needs to call setError() on.timestampstringISO 8601 timestamp of when the error occurred.pathstringThe request path that caused the error. Useful for logging.
Example: Nominee Validation Error (422)
{  "type": "INV-NM-01",  "title": "Nominee percentage validation failed",  "status": 422,  "detail": "All nominee percentage shares must sum to exactly 100%. Current total: 95%.",  "violations": [    { "field": "nominees[2].percentageShare", "message": "Shares must total 100%" }  ],  "timestamp": "2026-03-15T10:30:00Z",  "path": "/api/v1/member/nominees"}
HTTP Status Code Guide
StatusWhen to Use200 OKSuccessful GET or action with no new resource created.201 CreatedSuccessful POST that created a new record (registration, demise event).400 Bad RequestMalformed request — missing required fields, wrong format.401 UnauthorizedNo JWT token provided or token expired.403 ForbiddenValid JWT but insufficient role (e.g., member calling admin endpoint).404 Not FoundResource does not exist (e.g., unknown nesamId).409 ConflictBusiness rule conflict (e.g., user already has an active membership — INV-ML-02).422 Unprocessable EntityValidation failed (e.g., nominees do not sum to 100%, age ? 60). Always include violations array.500 Internal Server ErrorUnexpected server fault. Log full stack trace server-side; return generic message to client.

Section 1  ·  Home Page (Public Landing)
Route: /  |  domains/public/landing/pages/HomePage.tsx
The home page is the first thing any visitor sees — alumni, potential members, and returning members who are not yet logged in. Its job is to show that the program is real, active, and impactful. The ImpactStats component calls one backend endpoint to get live program metrics. Everything else on this page is static content served by the React app directly.

Frontend ViewModel: useHomeViewModel (or homeApi.getImpactStats()). The component mounts, fires this request, and populates the hero counters. If the backend is slow, the frontend shows placeholder skeleton loaders.

GET/public/impact-statsPublicFR-MA-19Return anonymised program-wide statistics for the public landing page.Why this API: The homepage ImpactStats component needs live numbers to build trust with visitors. These are fully anonymised — no member names, no individual amounts. This is what convinces an IRTTAA member to click "Register".Frontend integration: homeApi.getImpactStats() called on HomePage mount via useHomeViewModel.Invariants: None — no PII, no financial detail.Response Body (200/201)
FieldTypeDescriptiontotalMembersintegerCount of all memberships with current_status = ACTIVE. This is the "community size" number shown in the hero.familiesSupportedintegerCount of completed settlement_records rows. Shows how many families have received help.dfcEventsConductedintegerCount of distinct demise events (distinct reference_id batches in transaction_ledger with type = DEMAND).communitySupportInLakhsnumberSum of net_payout from settlement_records divided by 100000. Displayed as "?X Lakhs disbursed". Use decimal (e.g., 21.5).Note: Response requires no authentication. Backend query is a simple aggregate — cache it for 5 minutes if performance is a concern.

Section 2  ·  Login Page
Route: /login  |  domains/public/login/pages/LoginPage.tsx
Login uses the Spring Security One-Time Token (OTT) system — there are no passwords. The member enters their mobile number, receives an OTP via SMS (Twilio in production, console log in development), and exchanges it for a JWT. The JWT is stored in localStorage and auto-attached to every subsequent request by the Axios interceptor.

Frontend ViewModel: useLoginViewModel. Phase 1 collects the mobile number and calls ott/token. Phase 2 shows the OTP input and calls ott/login. On success, the JWT is written to localStorage and React Router pushes to /member.

PREREQUISITE: WebSecurityConfig.java must have generateTokenUrl("/api/v1/users/ott/token") added.Without this, Spring Security routes OTT generation to /ott/generate by default and CSRF blocks it.See swagger_api_testing_guide.md for the exact code change.
POST/users/ott/tokenPublicFR-MA-00Trigger OTT (One-Time Token) generation. Spring Security filter endpoint — not a @RestController method.Why this API: This is the "Send OTP" button. The member types their mobile number and taps send. Spring Security generates a magic-link token and dispatches it — in development, ConsoleOTTHandler prints it to the server terminal; in production, Twilio sends it as an SMS.Frontend integration: useLoginViewModel.sendOtp(mobileNumber) — Phase 1 of login flow.Invariants: None.
Request BodyFieldTypeRequiredDescriptionusernamestring (form)YesThe member's registered mobile number. Sent as application/x-www-form-urlencoded, not JSON. This is a Spring Security filter requirement.Note: Returns 200 OK with no body. The token is delivered out-of-band. If the mobile number is not found in user_profiles, still return 200 — never reveal whether a number is registered (security practice).
POST/users/ott/loginPublicFR-MA-00Exchange the raw OTT token string for a JWT. The frontend sends the token the member copied from SMS/console.Why this API: This is the "Verify OTP" step. The member enters the token they received. The backend validates it, looks up the associated user_profile, and returns a signed JWT that the frontend stores for all future requests.Frontend integration: useLoginViewModel.verifyOtp(token) — Phase 2 of login flow. On success, stores JWT in localStorage and navigates to /member.

Invariants: None.Request BodyFieldTypeRequiredDescription(body)raw stringYesThe raw OTT GUID from the SMS or console. Sent as plain text, not JSON. Example: ea11a3b1-abc1-1234-5678-abcdef012345
Response Body (200/201)FieldTypeDescription(body)stringRaw JWT token string. Frontend stores as-is in localStorage under key "nesam_token". The Axios interceptor reads this key and sends it as Authorization: Bearer <token> on every subsequent request.Note: The JWT must include at minimum: sub (user_uuid), role (MEMBER or ADMIN), and exp (expiry). The frontend does not decode the JWT for now — it only stores and forwards it.

Section 3  ·  Registration Page (8-Step Wizard)
Route: /register  |  domains/public/registration/pages/RegistrationPage.tsx
Registration is the most complex flow in the application. The frontend uses useRegistrationViewModel to drive an 8-step wizard. Data accumulates across steps and is submitted in one atomic transaction at the end. Three backend APIs power this flow: email verification (Step 1), OTP verification (Step 2), and the full submission (Steps 3–8).

Step order: ENTER_EMAIL ? ENTER_OTP ? CONTACT ? AGE ? NOMINEE ? MEMBERSHIP_BENEFIT ? SUMMARY ? PAYMENT ? PENDING_APPROVAL

POST/public/registration/verify-emailPublicFR-MA-01Step 1: Check if an email belongs to an IRTTAA alumni. Returns alumni data if found.Why this API: Before collecting any personal data, the system must confirm the applicant is a genuine IRTTAA member. This endpoint queries the parent association database. If the email is not found, registration is blocked. If found, the alumni data (name, batch, department, membership tier) is returned to pre-fill later steps.Frontend integration: registrationApi.verifyEmail({ email }) called from RegistrationEnterEmail component. The ViewModel stores the returned alumniData for use in fee calculation steps.Invariants: INV-EL-01: Only IRTTAA Life/Patron Members are eligible.
Request BodyFieldTypeRequiredDescriptionemailstringYesEmail address to look up in the IRTTAA parent association database.
Response Body (200/201)FieldTypeDescriptionstatusenumOne of: VERIFIED (found and eligible), NOT_FOUND (not in parent DB), PENDING (application already submitted awaiting approval), RESUME (incomplete draft exists).alumniData.namestringFull name from IRTTAA DB. Pre-fills the name field in Step 3 (contact details).alumniData.batchstringGraduation year (e.g. "2012"). Used in Step 4 (age) for slab optimisation for Patron and Founding members (FR-MA-05).alumniData.departmentstringBranch of study. Section 4  ·  Member Dashboard
Route: /member  |  domains/member/home/pages/MemberDashboardPage.tsx
The member dashboard is the authenticated home screen. It loads immediately after login and gives the member a complete at-a-glance view of their membership: who they are in the program, what their current status is, whether they owe a DFC contribution, and how many families they have helped. It also shows recent transactions for quick reference.
Frontend ViewModel: useMemberDashboardViewModel.fetchDashboard(). Called on component mount. The PrivateLayout wrapper ensures unauthenticated users are redirected to /login before this request is ever made.

Section 5  ·  Member Active DFC Page
Route: /member/active-dfc  |  domains/member/contributions/pages/MemberActiveDfcPage.tsx
This page is reached when the member has an open DFC contribution to make. It shows details of the event (who passed away, when, and what this member owes), and provides a payment button. For MVP, payment is a mock that marks the demand as SUCCESS.

GET/member/dfc/activeJWTFR-MA-13Return details of the currently open DFC event for the authenticated member.Why this API: The member needs to understand what they are paying and why.
